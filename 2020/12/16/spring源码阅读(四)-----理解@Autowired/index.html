annotation-config的解析

当遇到annotation-config标签的时候，会走到delegate.parseCustomElement(ele)。context命名空间下的标签是由ContextNamespaceHandler解析的。

找到annotation-config标签的解析类AnnotationConfigBeanDefinitionParser。解析的过程也比较简单，就是在AnnotationConfigUtils这个工具类中注册了ConfigurationClassPostProcessor，DefaultEventListenerFactory，EventListenerMethodProcessor，AutowiredAnnotationBeanPostProcessor，CommonAnnotationBeanPostProcessor，RequiredAnnotationBeanPostProcessor这6个BeanDefinition。其中ConfigurationClassPostProcessor，AutowiredAnnotationBeanPostProcessor，CommonAnnotationBeanPostProcessor，RequiredAnnotationBeanPostProcessor叫做后置处理器，也有称之为增强器。
spring中的后置处理器分为两种，一种是BeanFactoryPostProcessor，只有一个方法postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory)，主要是针对BeanFactory，在核心流程的第2步执行；另一种是BeanPostProcessor，包含postProcessBeforeInitialization(Object bean, String beanName)和postProcessAfterInitialization(Object bean, String beanName)，分别在核心流程的5和7步执行。
ConfigurationClassPostProcessor属于BeanFactoryPostProcessor，是spring体系中很重要的后置处理器，后面会详细讲解。
AutowiredAnnotationBeanPostProcessor，CommonAnnotationBeanPostProcessor，RequiredAnnotationBeanPostProcessor属于BeanPostProcessor。

类 接口 作用
ConfigurationClassPostProcessor BeanFactoryPostProcessor @Configuration @Component @ComponentScan @Import @ImportResource @Bean
AutowiredAnnotationBeanPostProcessor BeanPostProcessor @Autowired @Value @Inject @Lookup
RequiredAnnotationBeanPostProcessor BeanPostProcessor @Required
CommonAnnotationBeanPostProcessor BeanPostProcessor @PostConstruct @PreDestroy @Resource @EJB @WebServiceRef

Autowired生效

xmlBeanFactoryTest4运行起来是正常的，这个应该没什么疑问。为什么xmlBeanFactoryTest5不能通过测试呢？Autowired注入的SimpleBean为什么会出现空指针呢？classPathXml3为什么又没问题呢？如何再能让XmlBeanFactory里面的后置处理器和ClassPathXmlApplicationContext里面一样生效呢？
Autowired功能的实现主要依赖于AutowiredAnnotationBeanPostProcessor，它是一个BeanPostProcessor。核心流程图里面，BeanPostProcessor是在属性填充之后开始生效的。

BeanPostProcessor

AutowiredAnnotationBeanPostProcessor实现了很多接口，除了BeanPostProcessor，还有其子接口以及其它接口。

AutowiredAnnotationBeanPostProcessor几乎包含了整个BeanPostProcessor的继承体系。关于BeanPostProcessor的整个体系，本来是准备开单章详细讲解的。既然这里已经涉及到了，就简单说明一下。BeanPostProcessor的继承体系并不复杂，但是实现类比较多，看起来有些杂乱，我们只需要关注其子接口就可以了。

  1. BeanPostProcessor postProcessBeforeInitialization是bean在初始化之前需要调用的方法，postProcessAfterInitialization是bean在初始化之后需要调用的方法。postProcessBeforeInitialization和postProcessAfterInitialization方法被调用的时候。这个时候bean已经被实例化，并且所有该注入的属性都已经被注入，是一个完整的bean。
  2. InstantiationAwareBeanPostProcessor 在BeanPostProcessor的基础上增加了3个方法：postProcessBeforeInstantiation在Bean实例化之前调用,postProcessAfterInstantiation在Bean实例化之后调用，postProcessPropertyValues在属性填充之前调用。
  3. SmartInstantiationAwareBeanPostProcessor SmartInstantiationAwareBeanPostProcessor又在InstantiationAwareBeanPostProcessor的基础上增加了3个方法，都是辅助性方法：predictBeanType用于预测Bean的类型，返回第一个预测成功的Class类型，如果不能预测返回null，主要在BeanDefinition无法确定Bean类型的时候调用该方法来确定类型；determineCandidateConstructors方法用于选择合适的构造器，该方法在postProcessBeforeInstantiation方法和postProcessAfterInstantiation方法之间调用，如果postProcessBeforeInstantiation方法返回了一个新的实例代替了原本该生成的实例，那么该方法会被忽略；getEarlyBeanReference主要用于解决循环引用问题
  4. DestructionAwareBeanPostProcessor DestructionAwareBeanPostProcessor主要是容器中Bean被销毁之前调用，requiresDestruction判断是否需要销毁，postProcessBeforeDestruction执行销毁前的过程。
  5. MergedBeanDefinitionPostProcessor 该方法是bean在合并BeanDefinition之后调用。
    上述方法中，有两个词比较容易混淆：Instantiation表示实例化,对象还未生成；Initialization表示初始化，还未设置初始值。

AutowiredAnnotationBeanPostProcessor

AutowiredAnnotationBeanPostProcessor既实现了SmartInstantiationAwareBeanPostProcessor，也实现了MergedBeanDefinitionPostProcessor。理论上来说，AutowiredAnnotationBeanPostProcessor应该分别:
在实例化之前调用postProcessBeforeInstantiation
–> 实例化过程中调用determineConstructorsFromBeanPostProcessors找到构造器
–> 调用postProcessMergedBeanDefinition对合并后的RootBeanDefinition进行处理
–> 调用postProcessAfterInstantiation对实例化后的Bean进行处理
–> 调用postProcessPropertyValues对属性进行处理
–> 在初始化之前调用postProcessBeforeInitialization
–> 在初始化之后调用postProcessAfterInitialization

determineCandidateConstructors方法的功能我们已经说过了，就是为实例化对象找到合适的构造方法。这个方法其实并不复杂，不向spring中的某些方法，调用链几乎没有边界。这个方法的调用边界是可见的，核心逻辑就这么多。优先使用带有@Autowire注解并且require为true的构造器；次优先就是构造方法只有一个，并且有参数的构造器；再次就是主构造方法。
