概述

本章节，我们通过XmlBeanFactory来看看简单Bean是如何从BeanDefinition转化为Bean,并加载到容器中的。

BeanDefinition到Bean的转化

创建准备

我们先走测试1——xmlBeanFactoryTest1，看看简单Bean的转化过程。

必须在强调一次，spring中真正干活的方法大部分都以do开头，getBean最终在AbstractBeanFactory的doGetBean里面完成bean的转化。这里有关于spring里面一个容易与BeanFactory混淆的概念——FactoryBean，我们稍后会提到。另外，我们只看单例模式下的转化过程。

三级缓存

String beanName = getSingleton(String beanName, boolean allowEarlyReference)是为了防止重复创建单例Bean，同时也解决了循环依赖的问题。

引用核心流程

  1. BeanDefinitionReader将配置文件转化为BeanDefinition，并注册到BeanFactory
  2. BeanFactory后置处理器对所有的BeanDefinition进行处理
  3. 利用反射，实例化Bean
    实例化的过程，涉及到spring中的循环依赖问题。
  4. 填充各种属性和依赖
  5. Bean增强器进行增强处理：beafore
  6. 调用init()方法，对Bean初始化
  7. Bean增强器进行增强处理：after
  8. 完整的Bean注册到BeanFactory中

在核心流程中曾经提到过，实例化Bean之后，还会做很多处理。此处实例化但是还未完成后续处理的Bean会放在earlySingletonObjects。singletonFactories也和FactoryBean没有任何关系，里面存放的是ObjectFactory<?>。

这里的beforeSingletonCreation和afterSingletonCreation其实只是在singletonsCurrentlyInCreation做了添加和删除，最重要的还是singletonObject = singletonFactory.getObject()。singletonFactory.getObject()调用的是传进来的函数式接口的方法createBean(beanName, mbd, args)。

真正实例化

Object bean = resolveBeforeInstantiation(beanName, mbdToUse)涉及到InstantiationAwareBeanPostProcessor，它是BeanPostProcessor的子接口。关于BeanPostProcessor，我们后面统一讲解，此处暂且跳过。所以，此处的核心方法仍然是do开头的doCreateBean(beanName, mbdToUse, args)方法。

BeanWrapper可以看作是一个工具类，用来处理Bean的属性。instanceWrapper = createBeanInstance(beanName, mbd, args)撞见出来一个BeanWrapper。

obtainFromSupplier(instanceSupplier, beanName)与BeanDifinition的instanceSupplier有关，目前这个属性只有手动设置的方式。instantiateUsingFactoryMethod(beanName, mbd, args)与factoryMethodName有关，可以配置工厂方法来实例化Bean。determineConstructorsFromBeanPostProcessors(beanClass, beanName)与后置处理器有关，这里暂时用不上，暂且跳过。所以，上述代码在目前的配置下，最终只有instantiateBean(beanName, mbd)是实际产生作用的。

getInstantiationStrategy()得到的是一个CglibSubclassingInstantiationStrategy的实例。CglibSubclassingInstantiationStrategy是spring提供的实例化策略类，它的继承体系比较简单，大部分功能都在其父类SimpleInstantiationStrategy里面实现，涉及到方法重写，都是在CglibSubclassingInstantiationStrategy里面实现，比如ookup-method、replace-method。
beanInstance = getInstantiationStrategy().instantiate(mbd, beanName, this)是真正实例化的地方。instantiate和核心过程，就在核心流程图的最中间。然后把这个实例包装进BeanWrapper中，后面通过BeanWrapper来对Bean的属性进行操作。

applyMergedBeanDefinitionPostProcessors(mbd, beanType, beanName)涉及到MergedBeanDefinitionPostProcessor，也属于后置处理器，我们暂时设计不到。所以这里什么都没做，直接跳到addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean))吗，这里就涉及到spring的三级缓存和循环依赖了。在讲getSingleton(String beanName, boolean allowEarlyReference)时候，曾经提到过：

  • singletonObjects:用于保存BeanName和创建bean实例之间的关系，beanName–>bean Instance
  • earlySingletonObjects:也是保存BeanName和创建bean实例之间的关系，与singletonObjects的不同之处在于，当一个单例bean被放到这里面后，那么当bean还在创建过程中，就可以通过getBean方法获取到了，其目的是用来检测循环引用
  • singletonFactories:用于保存BeanName和创建bean的工厂之间的关系，banName–>ObjectFactory
  • registeredSingletons：用来保存当前所有已注册的bean

singletonFactories里面存储的ObjectFactory就是在addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean))此处添加的() -> getEarlyBeanReference(beanName, mbd, bean)。此时处于核心流程的34之间，刚好实例化但是还未处理属性，此时将BeanA的ObjectFactory放入缓存，当BeanB需要填充BeanA的时候，其实BeanA还未完成整个创建流程，但是BeanB可以认为BeanA已经创建成功，并从ObjectFactory<?>的方法中获取到BeanA的引用，并将此引用放入earlySingletonObjects中。后面所有依赖BeanA的实例都可以从earlySingletonObjects中获取到BeanA的实例。这里涉及到两个问题：1.为什么要用三级缓存，不用耳机缓存？明明二级缓存就够用了；2.代理类没有放入三级缓存中。我们在讲AOP的时候，会一一说明。

属性填充

实例化之后走到核心流程的第4步——属性填充populateBean(beanName, mbd, instanceWrapper)。xmlBeanFactoryTest1只是测试一个简单Bean，没有任何属性，这里我们要走xmlBeanFactoryTest2和xmlBeanFactoryTest3，看看两种情况下，属性填充的流程。关于自动注入的先跳过，这里没有使用到自动注入。

在Object resolvedValue = valueResolver.resolveValueIfNecessary(pv, originalValue)里面，会对SimpleBean2的属性simpleBean走一遍完成的实例化流程，然后通过BeanWrapper设置到Bean的属性中。BeanDefinitionValueResolver和BeanWrapper一样，可以看作是一个工具类，主要是将bean定义对象中包含的值解析为应用于目标bean实例的实际值，对外提供的只有一个方法resolveValueIfNecessary(Object argName, @Nullable Object value)。
当前我们要解析的value是一个RuntimeBeanReference，解析过程基本上和我们Bean的解析流程是一样的，就是重复Bean的实例化过程。

初始化方法

属性填充之后，进入核心流程的567步：

invokeAwareMethods(beanName, bean)涉及到Aware接口，我们暂时跳过。applyBeanPostProcessorsBeforeInitialization(wrappedBean, beanName)就是我们说的后置处理器BeanPostProcessor。它有两个方法，postProcessBeforeInitialization和postProcessAfterInitialization，分别在applyBeanPostProcessorsBeforeInitialization和applyBeanPostProcessorsAfterInitialization出调用，中间夹着invokeInitMethods(beanName, wrappedBean, mbd)。
对spring有一定了解，应该都知道初始化init方法。另外还有一个在init之前运行的afterPropertiesSet()方法，当你的Bean实现了InitializingBean接口，就需要实现这个方法。invokeInitMethods(beanName, wrappedBean, mbd)就是按照afterPropertiesSet->init这个顺序，分别运行这两个方法，实现Bean的初始化。
当这些过程全部完成之后，addSingleton(beanName, singletonObject)会将实例放入一级缓存registeredSingletons中，并删除二级三级缓存。这样一个不算复杂的Bean就实例化完成了。

比较一下xmlBeanFactoryTest2和xmlBeanFactoryTest3在属性填充上有什么区别，有助于较好的理解三级缓存的作用。
